---
title: "Benchmark"
description: "Measure Reckless search speed and verify engine consistency across a fixed set of positions."
---

The `bench` command runs a fixed-depth search over a curated set of 46 positions and reports the total node count and search speed. Its primary purpose is consistency checking — if a code change alters the node count, something about the engine's behavior has changed.

<Note>
  While `bench` gives a useful NPS figure, it is not a comprehensive benchmarking suite. Use it as a quick sanity check and a guide for PGO profiling, not as a definitive performance measurement.
</Note>

## Running bench

You can run `bench` in two ways.

### As a CLI argument

Pass `bench` directly when launching the engine:

```bash
reckless bench
```

This is the most common usage. The engine runs through all positions, prints a summary line, and exits.

### Inside a UCI session

Type `bench` at any point during an interactive UCI session:

```text
bench
```

When invoked this way, the output includes a full per-position breakdown with a formatted table.

## Arguments

`bench` accepts three optional positional arguments:

```bash
reckless bench [hash] [threads] [depth]
```

| Argument  | Default | Description                              |
| --------- | ------- | ---------------------------------------- |
| `hash`    | `16`    | Hash table size in MB                    |
| `threads` | `1`     | Number of search threads                 |
| `depth`   | `12`    | Search depth for each position (plies)   |

For example, to run bench with a 64 MB hash table, 4 threads, and depth 14:

```bash
reckless bench 64 4 14
```

## Output

### CLI mode output

When run as a CLI argument, bench prints a single summary line:

```text
Bench: 123456789 nodes 12345678 nps
```

This compact format is what tools like OpenBench and the PGO pipeline consume.

### UCI session output

When run inside a UCI session, bench prints a formatted table with one row per position, followed by a summary:

```text
--------------------------------------------------
          Nodes       Elapsed             NPS
--------------------------------------------------
  0    2847361        1.243s       2290248 N/s
  1    1923847        0.891s       2159201 N/s
  2    4102938        1.712s       2396576 N/s
  ...
--------------------------------------------------
    123456789       52.381s       2357123 N/s
--------------------------------------------------
```

Each row shows:
- **Index** — position number (0-based)
- **Nodes** — nodes searched at the given depth
- **Elapsed** — wall-clock time for that position
- **NPS** — nodes per second for that position

The final summary row shows cumulative totals across all 46 positions.

## What the numbers mean

| Field     | Meaning                                                               |
| --------- | --------------------------------------------------------------------- |
| **Nodes** | Total leaf and interior nodes visited during the search               |
| **Elapsed** | Wall-clock time from search start to `bestmove`                     |
| **NPS**   | Nodes per second — the primary measure of raw search throughput       |

The node count is the most important number for correctness verification. An identical binary run at the same depth should always produce the same node count for deterministic single-threaded runs.

## Why bench is useful

**Regression testing** — run bench before and after a change to confirm the node count is identical (for non-functional changes) or to measure how much search behavior changed.

**Performance comparison** — compare NPS between builds compiled with different flags or optimization levels to verify improvements.

**PGO profiling** — the PGO pipeline uses `cargo pgo run -- bench` to generate profile data. `bench` is the ideal workload because it exercises the search hot path without requiring external input.

<Tip>
  Use the default arguments when comparing across builds. Changing hash size or thread count will affect NPS measurements and make comparisons unreliable.
</Tip>
